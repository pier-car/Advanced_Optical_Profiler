self._session_controller.tolerances_changed.connect(

Il SessionController emette tolerances_changed(object) con un ToleranceLimits. Ma il MeasurementController che hai nel tuo codebase — ha davvero un metodo set_tolerance()? Questo metodo non è mai stato generato da me, e il tuo file originale potrebbe non averlo. Se manca → crash silenzioso o AttributeError al primo cambio tolleranze.

Verifica immediata:

Code
python -c "from controllers.measurement_controller import MeasurementController; print('set_tolerance' in dir(MeasurementController))"
B2 — _on_record_changed firma ambigua

In main_window.py:

Python
self._statistics_model.record_added.connect(self._on_record_changed)
self._statistics_model.record_removed.connect(
    lambda _: self._on_record_changed()

Il segnale record_added emette un oggetto (il record). Lo slot _on_record_changed è definito con _arg=None:

Questo funziona, MA: se il segnale record_added emette con una firma diversa (es. due argomenti, o nessuno), il collegamento fallisce silenziosamente in PySide6. Non crasha, semplicemente non viene chiamato.

🟡 RISCHI PROBABILI
R1 — _live_view._current_frame accesso diretto

In calibration_wizard.py e main_window.py:

Python
frame = self._live_view._current_frame
Accesso diretto a un attributo privato (_current_frame) che viene scritto dal thread del GrabWorker e letto dal main thread. Nessun lock. In pratica funziona perché Python ha il GIL e numpy.ndarray è un riferimento, ma se il GrabWorker sta scrivendo un nuovo frame mentre il main thread legge, potresti ottenere un frame parzialmente aggiornato (raro, ma possibile con array grandi).

R2 — _acquisition_controller._is_grabbing e ._grab_worker accessi privati

In main_window.py:

Python
if (self._acquisition_controller._is_grabbing
        and self._acquisition_controller._grab_worker is not None):
    self._acquisition_controller._grab_worker.frame_ready.connect(...)
Viola l'incapsulamento. Se l'AcquisitionController rinomina questi attributi → crash. Dovrebbe esporre is_grabbing come property e connect_frame_feed(slot) come metodo pubblico.

R3 — QDoubleSpinBox.valueChanged connesso a _on_nominal_changed

In TestSetupDialog, _on_nominal_changed modifica _spin_usl e _spin_lsl con blockSignals. Ma se blockSignals fallisce (errore logico, o viene chiamato da un altro thread), si crea un loop infinito di valueChanged. Il rischio è basso ma esiste.

1.2 Threading e Concorrenza
🔴 CRITICO
T1 — MetrologyEngine non è thread-safe

Il GrabWorker (thread secondario) chiama metrology_engine.process_frame(). La MainWindow (main thread) chiama metrology_engine.set_calibration() quando l'operatore completa la calibrazione. Se l'operatore calibra mentre il GrabWorker sta processando un frame:

Code
Thread GrabWorker:  process_frame() → legge self._scale_mm_per_px
Thread Main:        set_calibration() → scrive self._scale_mm_per_px
In Python con GIL il rischio di corruzione è basso, ma il risultato della misura in corso potrebbe usare metà vecchia calibrazione e metà nuova. Una misura potrebbe avere il top edge calcolato con la vecchia scala e il bottom edge con la nuova.

Mitigazione raccomandata: Aggiungere un threading.Lock in set_calibration() e process_frame(), o copiare i parametri di calibrazione in una struttura atomica.

T2 — StabilityDetector accesso concorrente

Lo StabilityDetector (se presente nel tuo AcquisitionController) mantiene un buffer circolare di misure. Il GrabWorker lo alimenta, ma il main thread potrebbe leggerlo (per il progress indicator). Se lo StabilityDetector non ha un lock interno, il buffer potrebbe essere in uno stato inconsistente durante la lettura.

T3 — StatisticsModel thread-safety

Il StatisticsModel viene scritto dal main thread (via Signal/Slot, che è safe), ma il get_snapshot() copia i dati. Se get_snapshot() viene chiamato da un thread secondario (improbabile ma possibile), il snapshot potrebbe essere inconsistente. Verifica che get_snapshot() sia chiamato solo dal main thread.

🟡 MEDIO
T4 — Timer di cleanup OSD nel LiveViewWidget

_cleanup_osd_messages() modifica self._osd_messages (lista) che viene letta da paintEvent. Entrambi sono nel main thread (QTimer + paintEvent), quindi è safe. MA: se qualcuno chiama show_osd_message() da un thread secondario, la lista viene modificata concorrentemente. Verifica che show_osd_message sia sempre chiamato dal main thread o via Signal.

1.3 Flusso Dati
🔴 CRITICO
D1 — SessionController non riceve dati anagrafici completi per il report

Il SessionController._offer_export() crea un ReportGenerator con:

Python
generator = ReportGenerator(
    operator_id=cfg.operator_id,
    session_start=session.started_at,
    calibration_scale=cfg.calibration_scale_mm_per_px,
)
Ma i dati anagrafici del prodotto (nome lotto, mescola, tipo bandina) sono nel session_name come stringa libera. Non c'è un campo strutturato per "Lotto", "Mescola", "Prodotto". Il report PDF mostra solo l'operatore e la data — non il nome della prova.

Fix: Aggiungere title=config.session_name nella chiamata a export_pdf().

D2 — quick_export() perde i timestamp

In SessionController.quick_export():

Python
records.append({
    "index": i,
    "timestamp": "",  # ← VUOTO!
    ...
})
L'export rapido (senza sessione) genera record senza timestamp. Il report PDF avrà la colonna Timestamp vuota per tutte le righe.

D3 — record_dicts non include tutti i campi usati dal report

Il MeasureRecord.to_dict() restituisce:

Python
{"index", "timestamp", "width_mm", "std_mm", "angle_deg",
 "n_scanlines", "status", "source", "is_valid"}
Ma il ReportGenerator.export_csv() cerca rec.get("n_scanlines", ""). Questo è coerente. Tuttavia il ReportGenerator.export_pdf() NON mostra n_scanlines nella tabella PDF — informazione persa nel report.

🟡 MEDIO
D4 — Calibration scale non propagata al SessionController all'avvio

In main_window.__init__(), il SessionController è creato con calibration_scale default 0.0:

Python
self._session_controller = SessionController(
    statistics_model=self._statistics_model,
    sessions_dir=SESSIONS_DIR,
    exports_dir=EXPORT_DIR,
    parent=self,
)
Il set_calibration_scale() viene chiamato solo in _on_new_session() e _on_calibration_done(). Se l'operatore apre una sessione senza ricalibrare e la calibrazione era pre-esistente, il SessionController ha cal_scale=0.0 → il report mostra "Calibrazione: 0.000000 mm/px".

Fix: Aggiungere nel costruttore di MainWindow, dopo la creazione del SessionController:

Python
if self._calibration_engine.is_calibrated:
    self._session_controller.set_calibration_scale(
        self._calibration_engine.scale_factor
    )
